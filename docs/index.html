Twenty-five years building software products as a UX/integration consultant — including fintech work at MarketAxess (electronic fixed-income trading), nine years of WPF / Silverlight enterprise consulting at Veracity Solutions, and a long stretch at IdentityMine literally bridging the gap between designers and developers. The last two years: a deliberate R&D program — ~83 experiments at varying stages of completion, aimed end-to-end at a single question: what does it actually take to direct AI through complete software builds — consistently, accurately, with minimal human intervention, and without inventing facts?

For those two years I haven't hand-written a line of code. Every line in the R&D program was generated by an LLM under my direction — orchestrator/worker patterns, custom Claude Code skills, deterministic spine with LLM at the leaves. The work is the orchestration: where the LLM can hold the architecture, where it can't, what review strategy each stack demands, and how to draw the boundary so the system stays correct.

The eight case studies below each pick one specific lesson from that R&D — what broke, what held, what each one taught about where AI delivers reliable work and where it breaks. A representative slice of the rest of the ~83 experiments follows, with live commit counts.

#02

Two Experiments in "Let the LLM Run It"

Two experiments pushing work directly onto the LLM at runtime: a 14-subagent bash orchestrator, and plain-English CLAUDE.md rules treated as constraints. Both hit hard walls. R&D finding: text instructions aren't enforcement boundaries, and runtime primitives need validation before architecture commits to them.

Lesson captured Read →

#03

Hallucination Needs Its Own Pipeline Stage

A 13-step LLM website-rebuild pipeline in TypeScript. Two production failure modes — content hallucination and structural drift between separately-generated sections — couldn't be closed by prompt engineering; each needed its own dedicated pipeline stage. R&D finding: pipeline length is diagnostic — every extra stage is a failure mode you discovered by running, not by designing.

TypeScript Puppeteer Postgres Claude API
Failure modes characterized Read →

#04

Personal AI Memory: The O(history) Problem

Six iterations of a long-running R&D track on AI assistants that accumulate memory across sessions (15-crate Rust workspace at the latest, 404 commits, plus a deployed Python instance). Two memory architectures explored, both with real costs. R&D finding: AI memory isn't a storage problem — it's a safety architecture problem. The local memory store got wiped multiple times in development before I added hooks to intercept irreversible commands.

#05

Active Research: Mixing Determinism and Non-Determinism

An open research thread on combining deterministic and non-deterministic execution so an LLM-powered system stays consistent and correct. Workflow tooling (pi-harness slash commands forcing explicit boundary decisions) plus empirical comparison (first-principles — RAG vs. open RLM vs. Lambda-RLM, same test suite). About to be field-tested in real-transactions Phase 2 — an AI real-estate transaction coordinator where uncontrolled LLM non-determinism isn't acceptable.

#06

AI Development Methodology at Scale

Two AI-orchestrated development methodologies — process constraints, slash commands, prompt templates. The 9-module TypeScript version (MAOD) finished cleanly. The 21-microservice Go version (MOAD) stalled at 1 service implemented. Same methodology, different module count. R&D finding: the ceiling isn't LLM capability — it's whether interface contracts between modules fit in the LLM's working context.

#07

Production AI Agents: Claude Agent SDK Stability as an Architectural Risk

A complete multi-tenant lead-response system for HVAC contractors: webhook receives a lead, AI agent qualifies it over SMS (Vonage), books on Google Calendar, live dashboard streams metrics. Backend, frontend, and infrastructure all built end-to-end. R&D findings: fast-moving AI SDKs need their docs bundled locally as a first-class dependency; and the stateful agent layer (core.py, 25.5KB) ends up the largest file in the backend — managing conversation state, retries, and tool-use constraints is more code than the business logic it serves.

Python FastAPI SvelteKit Claude Agent SDK
Reference implementation complete Read →

#08

Browser Automation Against Adversarial Targets

A procurement tool that fired browser-automation agents at electronics supplier portals (DigiKey, LCSC, Mouser, Arrow) in parallel, then ranked results with an LLM. Built end-to-end for the TinyFish Accelerator demo. Cloudflare bot protection broke agents non-deterministically — works-90%-then-silent-fail. R&D finding: browser automation against adversarial targets isn't productionizable; the pivot to Argus (same skill, public business-intelligence targets that want to be automated) was the correct move.

How I work

What I bring beyond the code

Translating between business and AI

Twenty-five years of consulting has been about bridging gaps — designer to developer, business stakeholder to engineer, customer goal to technical architecture. The AI work is the same job at a new layer: business problem to AI architecture, AI behavior to non-technical decision-maker. I default to plain language and keep jargon load-bearing.

Methodology built to be reused

The R&D program produced reusable methodology artifacts: MAOD/MOAD AI-orchestrated development methodologies, the Stoic Architecture workflow tooling (pi-harness slash commands), the canonical agent framework (caa-core@0.2.0). Plus ten years of independent training and expertise consulting before the AI program. Built to be transferred, not hoarded.

Findings, not finished products

None of the R&D builds have been deployed to sustained production with paying users — the goal is mapping how AI actually works, not landing a product. Test beds are real problem domains (real-estate transaction coordination, HVAC lead response, supplier-portal procurement, site rebuilds for small businesses), so what's learned transfers to real engineering, not toy demos. Each case study captures one specific finding about where AI delivers reliable work and where it breaks.
