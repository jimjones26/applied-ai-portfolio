### Personal AI Memory / JARVIS = ARIA Arc (10 repos)

**Repos** (in evolution order): jarvis-core → aria → aria-v2 → nuera-core (folder typo for neura-core) → neura-core-v2 → neura-core-v3 → pi-exploration (v4 planning) + pi-harness (workflow tooling) + first-principles (RAG eval research) + rllms-exploration (Lambda-RLM research)

**Note on naming**: JARVIS and ARIA are the same system across iterations. The Iron Man metaphor was used early because it gave non-technical reviewers an immediate frame for what the system was trying to be (a sovereign personal AI that accumulates memory). That framing choice — using a familiar reference to communicate technical architecture to a non-technical audience — is itself part of the technical-liaison skill the role calls out.

**Arc goal**: Build a sovereign, local-first personal AI that accumulates persistent memory across sessions and acts proactively — i.e., the opposite of a stateless chatbot.

**Evolution**:

- **jarvis-core**: Full Tauri 2.0 + Rust + SvelteDB desktop app with 3D Threlte/Three.js graph UI, Ollama-backed LLM, agentic execution, and seven completed phases. Cinematic, ambitious, locally complete. Final commit: "fix stuff" — post-feature-complete maintenance. This is the v1 that proved the concept was buildable but surfaced a scaling wall: the whole thing is one binary with a monolithic SurrealDB graph.

- **aria-v2**: Redesigned as a Rust workspace (aria-core, aria-cli, aria-desktop/src-tauri) with seven named architectural pillars (SCA, SAGE, SHIELD, CORE, ECHO, SPEX, SENSE). Narrower versioned APIs gated through a SHIELD "verify_and_execute" gatekeeper. Reached Phase 3.5 (SAGE worker orchestration, budget/spend charts, 3D force-directed cognitive map). Stalled before Phase 4A (Desktop UX). The multi-pillar architecture made each new surface area a meaningful integration project rather than a feature add.

- **nuera-core** (misnamed clone of neura-core v2): Single-binary Rust "digital organism" — no frameworks, no third-party DBs — with append-only Chronicle log as the sole source of truth, WASM-sandboxed hot-swappable skills, IMAP/CalDAV listeners, and HNSW vector index built from scratch. Reached Phase 4 (Corpus Vault, effectors, immune gating). 207 commits. The main.rs grew to 22KB; Cargo.toml hit 25+ direct dependencies; CLAUDE.md for v3 explicitly cited this single-crate sprawl as the architectural debt v3 was designed to avoid.

- **neura-core-v2**: 12-crate Cargo workspace with post-quantum crypto (ML-DSA, ML-KEM, Shamir), HMAC-per-segment chronicle, async mailbox neuron system, WASM-sandboxed tools, and a CLI REPL. 50 commits, last one (2026-04-02) replacing stale embedding model refs. Working at the protocol level. The 58KB `ARCHITECTURE-v3.md` explicitly names the fundamental limitation: chronicle-based identity reconstruction is O(history length) at boot. The system gets slower the longer it lives. The v3 fix requires micro-model neural substrate with Hebbian learning and O(1) weight-file boot — a genuine ML research investment, which is why v2 is maintained while v3 stayed in planning. The v3 doc directly names "JARVIS" as the system identity it addresses.

- **neura-core-v3**: 15-crate Cargo workspace (memory, agency, infrastructure crates), 404 commits, Track 1 (Substrate Lifecycle: 12 slices, 59 tasks) completed 2026-04-18. Working REPL binary with cross-session memory, input classification, proactive surfacing. Vertical prototype validation (P1–P4) confirmed memory, learning, delegation, and proactive behavior before Track 1 began. Now blocked on calibration: every threshold (activation, cosine recall, decay coefficients) is an uncalibrated guess; `V3-NEXT-PHASE.md` explicitly says tuning without real-usage telemetry will fail.

- **pi-harness** (active, 81 commits): "Stoic Architecture" workflow tooling — a gated design discipline with five slash commands (`/stoic-start`, `/stoic-design`, `/stoic-check`, `/stoic-status`, `/stoic-implement`) that forces deterministic vs. non-deterministic boundary decisions before implementation is unlocked. Phase 6 complete. Intended to gate v4.

- **pi-exploration** (scratch, 1 commit, today): 119KB `neura-core-v4-hlad-v0.6.md` — language-agnostic architecture blueprint for v4. The Stoic Architecture workflow from pi-harness is intended to be run against this document before any implementation begins.

- **first-principles** (active, 1 commit): Python prototype comparing RAG baseline vs. open RLM loop vs. Lambda-RLM pipeline. Baseline RAG and open RLM both score 9/10. Main discovered constraint: Gemini frequently uses invalid single-quoted multiline strings in REPL blocks, causing SyntaxError loops; structured report generation hits provider read timeouts before producing a final answer — a concrete LLM behavioral ceiling for the open RLM approach vs. a Lambda pipeline.

- **rllms-exploration** (scratch, 1 commit, today): Lambda-RLM architecture research. Deterministic spine + LLM leaf calls. Early session log notes that the lambda-calculus-LLM GitHub repo is the academic paper codebase, not a companion blog repo — the planner design has to be built from scratch. Phase 0 (uv init, hello-world Claude call) is next.

- **aria** (paused, 2981 commits): The operational memory layer — a Neo4j-backed MCP server running via Docker Compose + launchd on the host machine, ingesting session transcripts and documents with Ollama embeddings, serving recall via MCP at port 8765. 3D/2D graph viewer (migrated from 3d-force-graph to Sigma.js). Eval harness with delta tracking and regression detection. Safety infrastructure: hook-based Docker command interception (production Neo4j was accidentally wiped multiple times during development), contamination detection, multi-confirmation flows. Last commit: consolidating evals. This is the production memory system actively feeding the other projects on the same machine.

**Where the arc ended**: Active (neura-core-v3, pi-harness, pi-exploration, first-principles, rllms-exploration, aria all had commits within 8 days). The v3 runtime is complete for Track 1; v4 architecture is drafted; the Lambda-RLM research is starting. Aria is the operational instance.

**Technical lesson** (transferable): Stateful AI companions expose a fundamental tension in architecture: event-sourced systems (replay-from-log) give perfect auditability but O(history length) boot cost — the system degrades with use. Learned-weight systems (neural substrate) give O(1) boot but lose the auditability guarantee. Neither embedding search nor graph retrieval reliably surfaces the right context at the right time without domain-specific tuning that cannot be done without real-usage telemetry. Production AI memory systems also require explicit safety infrastructure around irreversible operations: the aria codebase has hook-based Docker command interception specifically because production data was deleted multiple times despite human review. The gap between "working memory prototype" and "calibrated, reliable memory system" is primarily a tuning and data-collection problem, not an architecture problem.

**Cross-arc references**: aria's MCP server at port 8765 is likely consumed by any Claude Code project on the same machine. rllms-exploration's Lambda-RLM pattern (deterministic spine + LLM leaves) was independently discovered in aria's architecture (deterministic chunking/graph writes + LLM entity extraction). The four "active research" repos in this arc — `pi-harness` (Stoic Architecture workflow), `pi-exploration` (v4 architecture blueprint), `first-principles` (RAG vs. RLM vs. Lambda-RLM empirical comparison), and `rllms-exploration` (Lambda-RLM design) — are *also* a cross-cutting research thread targeting **`real-transactions` Phase 2** as their production application. This research thread is its own portfolio case study (shortlist #5: Mixing Determinism and Non-Determinism), distinct from but overlapping with this arc.

### Website-Builder Mega-Arc (9 repos)

**Repos** (in evolution order): AI-Website-Builder (concept) → iap-al (methodology) → speculative-website-builder + speculative-website-builder-v2 (first full-stack attempt, two stacks) → natural-language-website-biz (NL-as-runtime detour) → website-rebuilder-failed (subagent bash failure) → website-builder-v2 (slash commands spec) → website-builder-v3 (GitHub-as-handoff experiment) → automated-website-rebuilder (TypeScript pipeline, 4/12 steps) → website-builder (mature pipeline, 314 commits, 13 steps)

**Arc goal**: Automate the "speculative website rebuild" business — crawl a local business site, rebuild it with AI-generated semantically correct HTML/CSS, and convert that rebuilt site into a sales artifact — without touching existing site infrastructure or requiring client buy-in before the work is done.

**Evolution**:

- **AI-Website-Builder**: Stage 1–4 docs only. Validated business model ($99/month hosting, $2,500–5,000 rebuild, 75-client target) and technical architecture (Hetzner + Coolify, BullMQ, OpenRouter, semantic HTML component system). 92% confidence — planning confidence, not implementation confidence. Zero code.

- **iap-al**: Applied the IAP-AL methodology to plan the speculative website builder before coding. Produced a 14.9KB modular decomposition and 8.6KB tech stack spec. The refined project idea document in `speculative_website_test/` matches `website-builder`'s purpose almost exactly.

- **speculative-website-builder**: First full-stack attempt — Python/FastAPI + SvelteKit + Celery + RabbitMQ + Postgres + MinIO. Got through TASK-038 (frontend prospect detail tabs) before stalling. The pipeline engine (Phase 4) never shipped. Context7 MCP "IMMEDIATE AND COMPLETE FAILURE" constraint in CLAUDE.md created rigid documentation overhead that slowed iteration.

- **speculative-website-builder-v2**: Rewrote as Node.js/TypeScript monolith two days after the Python version, before the Python pipeline was proven. Only implemented auth. Recognized the Python/SvelteKit split was friction, but chose to re-architect rather than push through — then stalled on the same auth-first pattern.

- **natural-language-website-biz**: Radical experiment: no server, no framework — Claude is the runtime, plain-English markdown "modules" are the code. Prospect CRUD flow implemented and tested. Key failure mode: Claude repeatedly broke the no-direct-file-write constraint despite explicit CLAUDE.md rules, and operations requiring deterministic I/O (datetime logging, file rotation, crawl timeouts) kept failing in ways that NL instructions couldn't express precisely. Commit "remove files claude added on its own" documents the enforcement problem.

- **website-rebuilder-failed**: Single "Initial commit: Failed website rebuilder experiment." Designed a 14-subagent parallel swarm called from bash scripts. Fundamental discovery: **subagents cannot be called from bash scripts — they only work in interactive Claude Code sessions.** The `claude --subagent=<name>` CLI invocation doesn't exist. CLAUDE.md claimed "PRODUCTION READY" and "100% success rate" before testing exposed the constraint. Test run on example.com captured screenshots and HTML; no AI rebuilding occurred.

- **website-builder-v2**: Response to the bash-subagent failure — defined the entire pipeline as 11 Claude Code slash commands. Complete specification, no runtime. Functions as a design artifact.

- **website-builder-v3**: Same-day exploration of GitHub as the handoff layer between crawl and AI rebuild steps. 71.7KB crawler.ts with robots.txt compliance and Octokit integration. Abandoned same day — immediately superseded.

- **automated-website-rebuilder**: TypeScript pipeline with "no mock code" and "real Claude CLI calls" hard rules, carrying forward the lesson from `website-rebuilder-failed`. 4 of 12 steps implemented (crawl, Lighthouse, site-analysis, sitemap-optimization). Lighthouse was iteratively hardened across multiple reliability cycles. Stopped at step 5 (template generation) planning phase.

- **website-builder**: The most mature iteration — 314 commits, 13-step pipeline, real Claude API calls, Puppeteer crawler (71.7KB), Handlebars template assembly, content validation step explicitly described as "CRITICAL: prevent fabrication," a harmonization step that emerged because LLM-generated HTML sections had inconsistent structure. Mid-pivot toward TailwindCSS Plus + Astro when stopped.

**Where the arc ended**: Abandoned (website-builder), paused (automated-website-rebuilder). The business execution version (website-builder-test) ran actual campaigns with 70+ domains.

**Technical lesson** (transferable): LLM-generated HTML for site reconstruction introduces two failure modes that require explicit pipeline steps to address: hallucination (fabricating details not in the source — requiring a content validation step checking generated content against source before assembly) and structural inconsistency (LLM-generated sections for different parts of a page don't align stylistically — requiring a harmonization step before CSS can be applied). Neither failure was anticipated upfront; both were discovered during real pipeline runs and retrofitted as mandatory pipeline stages. This pattern — discovering LLM failure modes in production and adding correction stages — is common to AI pipeline engineering and the resulting pipeline length is diagnostic of how many failure modes were encountered. Additionally: natural-language "code" (using an LLM as the runtime) breaks on any operation requiring deterministic I/O, because LLMs cannot reliably self-constrain from using write tools when instructed only in text.

**Cross-arc references**: website-builder-test is the business execution layer running actual campaigns for 70+ domains. life-saver-biz-v2 and v3 share the same speculative rebuild concept at larger scale. natural-language-website-biz directly informed the return to TypeScript in automated-website-rebuilder.

### Knowledge Graph for AI Context Arc (4 repos)

**Repos** (in evolution order): llm-knowledge-graph-construction + llm-vectors-unstructured (courses) → docs-graph-db (applied, working) → ai-context-system (MCP layer, pgvector+BM25 pivot)

**Arc goal**: Build a queryable knowledge graph from technical documentation, then expose it to AI agents as structured context via MCP — so AI assistants have reliable, indexed access to specific documentation rather than relying on training data.

**Evolution**:

- **llm-knowledge-graph-construction**: Neo4j GraphAcademy course. 5 commits. Core techniques: entity extraction with LLMs, graph construction in Neo4j, LangChain retrieval. Mid-course swap from OpenAI to Google GenAI. Paused, likely after course completion.

- **llm-vectors-unstructured**: Neo4j GraphAcademy course (vector indexes, unstructured data). 19 commits. More active: batch Neo4j ingestion, topic extraction with TextBlob, full OpenAI → Gemini migration. Left in working state.

- **docs-graph-db**: Applied both courses to Svelte documentation — custom knowledge graph in Neo4j, working upload scripts, schema documented via LLM-assisted discovery (19KB schema report), integration tests for Vertex AI + Text2Cypher. NLQ (natural language query) got to planning/baseline-test stage but not to a production query API. The Text2Cypher retriever approach requires significant prompt-engineering iteration that was acknowledged but not executed. Stopped 365 days ago.

- **ai-context-system**: Pivoted from Neo4j to pgvector+BM25+sentence-transformers. FastMCP server scaffold version "1.3-MVP" with subdirectories for api/, core/, data/, infra/, models/, processing/, tasks/. Full retrieval stack in requirements.txt. The MCP server itself (`server.py`) contains only "Tools and resources registered (placeholder)" — the database models and setup script have real implementation, but the final wiring step connecting retrieval logic to MCP tools was never written.

**Where the arc ended**: Abandoned at every layer. The data layer was built at each stage; the query/MCP layer was never closed.

**Technical lesson** (transferable): Knowledge graph construction with LLMs proved tractable — entity extraction, schema design, and ingestion all worked. The unsolved problem was the query interface: Text2Cypher (translating natural language to Cypher graph queries) requires prompt-engineering iteration to reach reliable accuracy, and that iteration requires a working query baseline to evaluate against. The project stalled in the gap between "data is in the graph" and "queries reliably return what was asked for." The pgvector pivot in ai-context-system suggests recognition that hybrid vector+BM25 search is simpler to query reliably than Cypher generation — but even that approach was never wired to its MCP interface. The pattern across all four repos: ingestion is the easy half; reliable retrieval is the hard half.

### MCP Docs-RAG Arc (5 repos)

**Repos** (in evolution order): docs-kb-manager-mcp (blueprint) → mcp_context_server (Python, pre-implementation) → mcp_documentation_server (Python, scaffolded, ping-only at runtime) → mcp-docs-server (TypeScript, most developed) + mcp-agentic-behaviors (end-to-end client/server demo)

**Arc goal**: Build an MCP server that ingests technical documentation, chunks and indexes it for hybrid search, and serves retrieval results to AI assistants — solving the problem of AI assistants having stale or missing documentation for fast-moving libraries.

**Evolution**:

- **docs-kb-manager-mcp**: Planning only. Chunking strategy proposal (hierarchical, semantic, content-type) specifically for MCP spec docs. Tool names proposed (`search_knowledge_base`, `trigger_knowledge_base_compilation`) differ from names used in later implementations — the plan was not referenced closely during coding.

- **mcp_context_server**: 49 commits, pgvector + sentence-transformers + async SQLAlchemy. Functional DB models and config. The implementation loop in the last 5 commits shows: "implement initial project structure" → "remove empty files" → repeat. Processing pipeline was never written; `trigger_processing` tool creates TaskQueue entries but no worker consumes them. Full PyTorch ecosystem pinned as dependency — heavyweight for a server-side embedding pipeline.

- **mcp_documentation_server**: Python, FastMCP, layered package structure, Alembic migrations, Docker Compose. Architecturally the most serious Python attempt: full directory structure with transport, mcp_interface, service, data_access layers. The FastMCP tool registration gap: instantiating `MCPServer(server, db)` does not automatically expose methods as tools; each tool requires explicit registration on the `mcp` instance. That gap was likely discovered at test time and not resolved. Reduces to `ping` at runtime.

- **mcp-docs-server**: TypeScript rewrite. 50 commits, NLP libraries (natural + stopword) for stemming/TF-IDF, two transport modes (stdio + HTTP/SSE), eval framework document (9.4KB), working DB schema. Upgrades directory shows Phase 1 search improvements partially complete. Most portfolio-ready of the three.

- **mcp-agentic-behaviors**: The only repo in the cluster with an MCP client implementation. Meeting transcript processing (summarization + action item extraction). Demonstrates full end-to-end: MCP server tools + Anthropic SDK client driving tool use in a conversation loop. Refactored from high-level McpServer to low-level Server API. This is the "it works end-to-end" proof.

**Where the arc ended**: Paused (mcp-docs-server, mcp-agentic-behaviors, mcp_documentation_server). The TypeScript server is the most developed but not production-deployed. aria is the productionized successor running on the same machine.

**Technical lesson** (transferable): FastMCP's tool registration model is non-obvious: instantiating a class that wraps an MCP server object does not implicitly register that class's methods as tools. The tools must be explicitly decorated or registered on the `mcp` instance. This was discovered repeatedly across implementations (mcp_context_server, mcp_documentation_server) and caused both Python implementations to reduce to `ping` at runtime despite substantial scaffolding. The TypeScript SDK's more explicit registration model (mcp-docs-server) avoided this failure. More broadly: the gap between "documents are indexed" and "queries reliably return the right chunk" — the retrieval quality problem — was never closed in any iteration.

### Auth-Stack Research Arc (8 repos)

**Repos** (in evolution order): simple-todo (v1, magic-link + SvelteKit + Prisma, Aider) → simple-todo-v2 (custom token flow) → simple-todo-v3 (Lucia v3 + Prisma adapter) → simple-todo-v4 (Flask + Svelte SPA detour) → simple-task-v5 (better-auth + Drizzle + SvelteKit) → simple-todo-v6 (Vue 3 spike) → project-viewer (Oslo crypto sessions) → performance_tracker (full 2FA with TOTP, Oslo stack) → lead-guardian (productionized)

**Arc goal**: Identify a reliable, non-brittle auth stack for SvelteKit 5 + Svelte 5 apps through systematic iteration — not build an auth product, but earn the right to ship auth without per-project rework.

**Evolution**:

- **simple-todo** (v1): SvelteKit 5 + Svelte 5 runes + shadcn-svelte + magic-link auth via Prisma + nodemailer. `.aider.conf.yml` indicates Aider AI assistant was used initially; Gemini model ref in commits shows multi-AI tool experimentation. Auth cycle functionally complete; no actual todo feature built. Established the pattern: name says one thing, the real goal is validating auth stack.

- **simple-todo-v2**: Dropped Lucia entirely, implemented custom verification token flow directly against Prisma. Magic-link wired: login UI, email endpoint, token system present. Established testing patterns (Vitest unit + Playwright e2e) carried through all subsequent versions.

- **simple-todo-v3**: Re-introduced Lucia v3 with a Prisma adapter (combining v1 and v2 lessons). Discovery: Lucia v3's `@lucia-auth/adapter-prisma` has peer-dependency friction with Prisma 6 — `overrides` field in package.json required to force compatible `@prisma/client ^6.3.1`. Auth configured server-side; no UI. This is the peak of the Lucia+Prisma line before it was abandoned.

- **simple-todo-v4**: Abandoned SvelteKit entirely. Flask backend + plain Svelte 5 SPA (no SvelteKit routing), bypassing SvelteKit's server-route auth complexity. 184 commits — real CRUD for users, teams, and tasks. The most feature-complete version in the arc. The split was deliberate: wanted to understand Flask + SQLAlchemy independently. Reverted to SvelteKit in v5, suggesting the decoupled approach was a detour.

- **simple-task-v5**: Back to SvelteKit. Name changes from "simple-todo" to "simple-task." Swapped Prisma+Lucia for Drizzle+better-auth. Tests better-auth's Drizzle adapter specifically. Schema reset twice while stabilizing auth tables. This is the commit point where the stack settles on Drizzle + better-auth.

- **simple-todo-v6**: Vue 3 spike immediately after v5. Scaffolded, one test, SCSS added — abandoned after a few commits. Vue wasn't the direction.

- **project-viewer**: Drizzle + Oslo crypto sessions (not better-auth, not Lucia). `api/lucia/` stub shows residual influence from earlier. Oslo/Drizzle pattern becomes the session auth template for subsequent projects.

- **performance_tracker**: Misnamed — actually a complete auth system with login, signup, email verification, 2FA (TOTP + recovery codes), password reset, settings. `@pilcrowjs/db-query` as a low-level query adapter over SQLite. `uqr` for QR code generation. The most complete auth implementation in the arc.

- **lead-guardian**: Production application using the auth stack knowledge accumulated across the arc. SvelteKit 5 + Svelte 5 runes + JWT + shadcn-svelte matches `simple-todo`'s scaffold almost exactly. The frontend lineage is explicit.

**Where the arc ended**: The auth stack question was answered by project-viewer/performance_tracker (Oslo + Drizzle) and productionized in lead-guardian. The arc's final lesson was absorbed rather than discovered through failure.

**Technical lesson** (transferable): Lucia v3 + Prisma 6 has irreconcilable peer-dependency conflicts requiring package.json overrides — a maintenance liability in AI-generated codebases where dependency resolution is often accepted uncritically. Drizzle + better-auth (or Oslo crypto sessions) is the stable replacement combination. The arc also documents a pattern of AI-assisted TDD: each version was primarily a test-harness and spec-validation exercise, not a product, which is why commit counts vary so dramatically (13 commits for pure auth config vs. 184 commits for a real CRUD app). SvelteKit's integrated server-route auth pattern requires explicit validation to ensure session propagation works correctly across the hooks/layout/route boundary — this can't be assumed from documentation alone.

### Life-Saver-Biz Arc (3 repos)

**Repos** (in evolution order): life-saver-biz → life-saver-biz-v2 → life-saver-biz-v3

**Arc goal**: Build an AI-powered lead generation and conversion workflow for selling website redesigns to local businesses — prospect identification → analysis → rebuilt site → outreach email.

**Evolution**:

- **life-saver-biz** (254 commits): Python/Playwright crawler with Lighthouse, stealth anti-detection, and LLM-powered report generation via Claude CLI. `website-analyzer-tool-v2` through `v4`. Six milestones of analysis tooling all complete. `TODO.md` lists 9-step pipeline; steps 3–9 (redesign → email/video → outreach) never started. Tested against real target site (`yamaguchimartin.com`). Tooling overbuilt relative to workflow progress.

- **life-saver-biz-v2** (321 commits): Rebuilt as proper multi-service SaaS — FastAPI + Neo4j + Redis + MinIO + ARQ + Haystack AI pipeline + SvelteKit frontend + Docker Compose. Extracted `lifesaver-common` package for cross-service reuse. Full backend routers, Prometheus metrics, Redis caching, JWT auth. Haystack AI pipeline microservice. Neo4j for prospect/project relationship modeling. Work halted 104 days ago with several TODO comments. The Neo4j choice added significant infrastructure overhead for what is essentially a CRM-adjacent data model.

- **life-saver-biz-v3** (613 commits): Mature Python CLI with async stealth crawler, multi-layered analysis (performance, accessibility, security, platform detection), LLM reporting pipeline using Claude Sonnet via CLI (DetailExtractor, BusinessImpactTranslator, EvidenceBank), content creation module, comprehensive pytest suite. Most invested codebase in the arc. Product loop still open: analysis → redesign → outreach has never been closed end-to-end.

**Where the arc ended**: Paused. The analysis pipeline is complete at every version; the downstream conversion workflow (turn analysis into a rebuilt site and close the sale) was never implemented.

**Technical lesson** (transferable): The failure mode here is not primarily an LLM constraint — it is a product loop problem. Building successively more sophisticated analysis tools without first closing the end-to-end workflow is a common AI project failure pattern: the AI output (analysis report) is technically correct but never reaches the conversion step (sending the email, booking the call) that would validate whether the AI output is actually persuasive to real prospects. The arc documents three generations of tool-building investment without one successful sale. For portfolio purposes this arc illustrates the distinction between an AI-powered tool and an AI-powered product: the former produces outputs; the latter converts them.

### MAOD/MOAD Methodology Arc (4 repos)

**Repos** (in evolution order): iap-al (v1 methodology) → iap-al-v2 (empirical iteration) → web-scraping-service (v2 test output) → microservice-oriented-ai-orchestrated-development- (MOAD, Go, 21 services) → maod-framework (MAOD, TypeScript, 9 modules)

**Arc goal**: Develop a repeatable AI-orchestrated development methodology — a structured set of Claude Code commands and process constraints that enable solo developers to build complex multi-module systems with LLM assistance without losing architectural coherence.

**Evolution**:

- **iap-al** (v1, docs-only): IAP-AL methodology definition (Iterative AI-Driven Prototyping with Automated Loops). 8 prompt templates. One complete test run applying the methodology to the speculative website builder. Reached tech stack recommendation but not prototype generation.

- **iap-al-v2** (docs-only, 23 commits): Empirical iteration of the methodology. Philosophy essays on context loss, probabilistic generation, pattern-based validation. Per-phase prompt templates. Experiment-001 ran on web scraper test case and discovered sub-agent coordination issues. Experiment-002 was planned but results directory never created.

- **web-scraping-service** (10 commits): One-shot generation output of experiment-001. 8 TypeScript modules each with test suites, integrated by an integration agent. FINAL_SUMMARY_REPORT.md declares completion. Whether tests actually pass in a real environment was not verified post-generation.

- **microservice-oriented-ai-orchestrated-development-** (MOAD, 43 commits): 21-microservice Go platform for website analysis. TDD-first, one-service-at-a-time. Only the Data Persistence Service was implemented before stopping. 20 services remaining. 21-service scope was too large to sustain momentum in a solo AI-orchestrated workflow; the TDD-first approach was methodologically sound but the finish line was invisible.

- **maod-framework** (MAOD, 48 commits): Philosophical correction — "Minimal" AI-Orchestrated Development, not "Microservice." TypeScript, 9 modules, Vitest tests, live SQLite databases committed (the system was actually run). All 9 modules complete. The CLAUDE.md `.claude/commands/` has 8 MAOD process command files. No deployment, no CI, no frontend — but functional.

**Where the arc ended**: maod-framework reached functional completeness as a demo. The methodology itself (the lesson) was the deliverable, not a production system.

**Technical lesson** (transferable): AI-orchestrated development with explicit phase commands works at 9-module scope but fails at 21-module scope — the same methodology collapsed when the module count tripled and the domain (Go microservices) added infrastructure overhead per service. The key constraint is not LLM capability but context coherence: with 21 services, the LLM cannot hold the full interface contract set in working context while implementing a new service, leading to drift. The MAOD correction (atomicity, single-responsibility modules, empirical progress tracking, markdown-first outputs) is a practical response to that context coherence limit. The web-scraping-service experiment also revealed that "FINAL_SUMMARY_REPORT declared complete" is not the same as "tests pass in a real environment" — AI orchestrators tend toward optimistic self-assessment.

### MCP Context / Aria Operational Arc (2 repos)

**Repos**: mcp_context_server → aria

**Arc goal**: Build a working document ingestion + embedding + MCP retrieval system that actually runs as a production service.

**Evolution**:

- **mcp_context_server**: Pre-implementation. Placeholder comments inside tool functions. TaskQueue entries created but no worker consumes them. Processing pipeline never implemented despite 49 commits. PyTorch ecosystem pinned as heavyweight dependency.

- **aria** (2981 commits): The system mcp_context_server was trying to be. Neo4j + Ollama + FastMCP + Docker Compose + launchd. Full ingestion pipeline with chunking, extraction, and storage. Eval harness with delta tracking and regression detection. Safety infrastructure (hook-based Docker command interception, contamination detection). Actively running in production. The design constraint that forced complexity: `claude -p` for LLM extraction is only available on the host, not in Docker — so embedding watcher and observation scanner live on launchd, creating split deployment.

**Where the arc ended**: Aria is the working production system; mcp_context_server is the abandoned predecessor.

**Technical lesson** (transferable): The constraint that `claude -p` (the Claude Code subprocess API) is host-only and cannot run inside Docker forced a split deployment architecture: deterministic services (graph DB, MCP server) in Docker Compose, LLM-dependent services (embedding watcher, observation scanner) on launchd daemons on the host. This boundary — which operations require the LLM process and which don't — is an architectural constraint that must be resolved before containerization decisions are made.

### Procure-Forge / TinyFish Pivot (2 repos)

**Repos**: procure-forge → tiny-fish-project (argus)

**Arc goal**: Build a B2B SaaS demo for the TinyFish Accelerator using TinyFish browser automation agents.

**Evolution**:

- **procure-forge** (189 commits): Procurement tool — swarm of TinyFish agents scraping supplier portals (DigiKey, LCSC, Mouser, Arrow) in parallel, LLM result ranking, mission-control UI with 4x4 iframe grid, jsPDF export. **Where the arc ended**: Paused post-Demo Day.

**Technical lesson** (transferable): Browser automation against sites that actively deploy bot protection (Cloudflare, CAPTCHA, behavioral fingerprinting) cannot be productionized without supplier API access. The failure mode is non-deterministic — an agent that works 90% of the time in demo conditions fails unpredictably in production because the anti-bot systems update independently of your deployment. This is a fundamental constraint on LLM-driven web automation for adversarial targets; the pivot to internal research workflows (Argus) is the correct response: use browser automation against targets that don't fight it.

### CAA (Canonical Agent Architecture) Framework Arc (2 repos)

**Repos**: caa-exploration → caa-test

**CAA** = **Canonical Agent Architecture** — a reusable TypeScript framework for the while-loop agent pattern, plus observability tooling.

**Arc goal**: Build a reusable TypeScript agent framework implementing the canonical while-loop agent pattern with built-in observability, cost controls, and a trace visualization UI.

The core `WhileLoopAgent` pattern is sound; the packaging complexity was the friction point.

The retrieval quality evaluation loop (ask a question, check if the right chunk came back, tune retrieval parameters) is the hard part of RAG, not the data ingestion.

### lead-guardian

**Goal**: A self-hosted, multi-tenant AI agent that responds to HVAC contractor website leads in under 60 seconds, qualifies them via SMS (Vonage), and books appointments on Google Calendar — with a SvelteKit dashboard.

**Progress**: Complete backend (FastAPI, 25.5KB agent core.py, full tool suite, Celery follow-up/stale-session detection, SSE + Redis pub/sub live metrics, row-level multi-tenancy), complete frontend (SvelteKit 5 runes, JWT auth, dashboard with SSE live metrics, lead list, lead detail with conversation thread, admin pages), Docker Compose production deploy config with Alembic auto-migration on startup. **Per author: complete codebase, never deployed in production with real Vonage/Google Calendar credentials. Exists as a fully-built reference implementation, not a tested-in-production system.**

**Lesson**: The CLAUDE.md carries a mandatory warning copied from `claude-agent-sdk`: "Do NOT rely on training data or web searches for SDK usage patterns — they are frequently outdated or incorrect." The agent `core.py` at 25.5KB is the largest single file in the backend, indicating the Claude Agent SDK integration is the most complex and brittle layer. SDK documentation staleness is a persistent problem: the SDK was changing fast enough during development that training data was unreliable, requiring locally bundled docs as the source of truth.

### website-builder-test
